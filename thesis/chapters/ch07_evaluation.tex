\chapter{Evaluation}
\label{ch:evaluation}

This chapter presents the empirical evaluation of \versimux{}. We define the research hypotheses, describe the experimental setup, present the results, and analyse the findings against each research question.


% ---- 7.1 Research Questions and Hypotheses -----------------------------------
\section{Research Questions and Hypotheses}
\label{sec:hypotheses}

To evaluate the capabilities of \versimux{} as an automated web usability and accessibility repair system, we formulate four research questions (\acr{RQ}s) that correspond to the three gaps identified in \cref{sec:motivation}:

\begin{description}
  \item[RQ1: Diagnostic Accuracy.] How does \versimux{}'s diagnostic accuracy compare to expert manual evaluation and standard automated baselines?
  \item[RQ2: Grounding and Fidelity.] Do the grounding guards and trace verification mechanisms effectively prevent action hallucinations and maintain simulation coherence?
  \item[RQ3: Patch Quality.] Are the generated code patches syntactically valid, applicable, and effective at resolving targeted usability and accessibility failures?
  \item[RQ4: Developer Perception.] Do frontend developers perceive the generated reports and code patches as actionable, trustworthy, and useful compared to standard tools?
\end{description}

\Cref{tab:hypotheses} defines the research questions and directional hypotheses.

\begin{table}[htbp]
  \centering
  \caption{Research questions and directional hypotheses.}
  \label{tab:hypotheses}
  \begin{tabularx}{\textwidth}{@{}c L{5cm} X@{}}
    \toprule
    \textbf{ID} & \textbf{Research Question} & \textbf{Hypothesis} \\
    \midrule
    RQ1 & How does \versimux{}'s diagnostic accuracy compare to expert manual evaluation? & H1: \versimux{} achieves $\geq$70\% recall and $\geq$60\% precision against expert-identified issues. \\
    \addlinespace
    RQ2 & Do grounding guards and trace verification effectively prevent hallucinated diagnoses? & H2: Grounding guards block $\geq$30\% of attempted actions as hallucinated selectors. Trace verification discards $\leq$20\% of traces while retaining $\geq$80\% of genuine issues. \\
    \addlinespace
    RQ3 & Are the generated patches syntactically valid, semantically correct, and applicable? & H3: $\geq$75\% of patches apply successfully. $\geq$60\% resolve their target issues as confirmed by verification. \\
    \addlinespace
    RQ4 & Do developers perceive reports as actionable and trustworthy? & H4: Developers rate \versimux{} reports as more actionable than static checker reports. \\
    \bottomrule
  \end{tabularx}
\end{table}


% ---- 7.2 Experimental Setup -------------------------------------------------
\section{Experimental Setup}
\label{sec:experimental-setup}

\subsection{Test Pages}
\label{sec:test-pages}

The evaluation was conducted on four web interfaces developed to represent typical interactive components in modern web applications. Each page contains planted usability and accessibility violations representing WCAG 2.2 Level A and AA criteria, alongside common interaction friction points. \Cref{tab:test-pages-spec} details the properties of these test pages.

\begin{table}[htbp]
  \centering
  \caption{Specification of Test Pages used for Evaluation.}
  \label{tab:test-pages-spec}
  \begin{tabularx}{\textwidth}{@{}l c c c X@{}}
    \toprule
    \textbf{Test Page} & \textbf{DOM Nodes} & \textbf{Interactive Nodes} & \textbf{Planted Failures} & \textbf{Primary Focus Areas} \\
    \midrule
    Login Form         & 42                 & 6                          & 8                         & Missing labels, outline suppression, poor contrast. \\
    Product Catalog    & 186                & 24                         & 14                        & Custom dropdown dropdowns, keyboard focus traps. \\
    Data Dashboard     & 310                & 38                         & 16                        & Missing live region updates, custom SVG button attributes. \\
    Registration Wizard& 145                & 15                         & 10                        & Tab sequence order, dynamic form validation notifications. \\
    \bottomrule
  \end{tabularx}
\end{table}

\begin{itemize}
  \item \textbf{Login Form:} A standard authentication interface. Planted errors include text input elements lacking programmatic labels, color contrast ratios below WCAG AA thresholds on validation error texts, and the explicit CSS override \code{outline: none} on input focus.
  \item \textbf{Product Catalog:} An interactive grid featuring filters, sorting controls, and modal detail views. Planted errors include filter buttons lacking alt text, custom selection elements styled as un-focusable \code{<div>} wrappers, and a product-details modal that does not trap keyboard focus.
  \item \textbf{Data Dashboard:} A complex visual interface featuring dynamic filtering, charts, and table pagination. Planted errors include charts lacking screen-reader accessible data tables, dynamic status message boxes lacking \code{aria-live} notifications, and SVG icons lacking labels.
  \item \textbf{Registration Wizard:} A multi-step stepper control. Planted errors include step containers placed out of logical tab sequence, keyboard focus traps within form inputs, and validation alerts that do not receive programmatic focus.
\end{itemize}

\subsection{System Configuration}
\label{sec:system-config}

The system was configured with the following model parameters:
\begin{itemize}
    \item \textbf{Supervisor, Resolver, and Verifier Nodes:} Routed to \code{llama-3.3-70b-versatile} via Groq API. The temperature was set to 0.2 to ensure deterministic reasoning, with a maximum token limit of 4096.
    \item \textbf{Persona Simulation Nodes:} Routed to \code{llama-3.1-8b-instant} via Groq API. The temperature was set to 0.7 to support diverse exploratory path generation, with a maximum token limit of 2048.
    \item \textbf{Persona Configurations:} For each test page, the supervisor generated a swarm of 4 user personas representing visual impairments (screen-reader users), motor limitations (keyboard-only users), color blindness (Protanopia), and cognitive constraints (non-technical users).
\end{itemize}

\subsection{Hardware and Environment}
\label{sec:hardware-env}

Evaluations were executed on a dedicated system running Windows 11 Pro, equipped with an AMD Ryzen 9 5900X CPU (12 cores, 24 threads), 64 GB DDR4 RAM (3600 MHz), and a 1 TB NVMe SSD. The browser automation layer used Playwright v1.40 running Chromium in headless mode, driven by Python 3.11.2.

\subsection{Repetitions and Non-Determinism}
\label{sec:repetitions}

Because large language models exhibit stochastic behavior even at low temperatures, each test page was processed through 5 independent end-to-end runs. All reported metrics represent the arithmetic mean across these repetitions, with standard deviations reported where appropriate.


% ---- 7.3 Expert Ground Truth Construction ------------------------------------
\section{Expert Ground Truth Construction}
\label{sec:ground-truth}

To establish a baseline for diagnostic accuracy, a panel of three web accessibility consultants (each holding IAAP WAS or CPACC credentials) independently audited the four test pages. The experts conducted a manual audit using assistive technology (NVDA screen reader, keyboard-only walkthroughs, and screen magnification software) along with developer tools to inspect DOM attributes.

The audits identified a combined total of 48 distinct usability and accessibility issues across the four interfaces. Inter-rater reliability was measured using Fleiss' Kappa, yielding $\kappa = 0.84$, indicating strong agreement among the auditors. Discrepancies were resolved through a consensus review. This consolidated list of 48 issues serves as the ground truth for our diagnostic accuracy evaluations.


% ---- 7.4 Diagnostic Accuracy (RQ1) -----------------------------------------
\section{Diagnostic Accuracy}
\label{sec:rq1-results}

We evaluated \versimux{}'s diagnostic accuracy against the expert ground truth and two automated baselines:
\begin{enumerate}
  \item \textbf{axe-core (v4.8.2):} The industry-standard static accessibility rules library, executed inside Playwright.
  \item \textbf{Google Lighthouse (v11.4.0):} The automated audit utility built into Chrome, which wraps axe-core and includes additional contrast check utilities.
\end{enumerate}

\Cref{tab:rq1-results} presents the precision, recall, and $F_1$-score for each condition, representing the aggregated performance across all four test pages.

\begin{table}[htbp]
  \centering
  \caption{Diagnostic accuracy: \versimux{} vs. automated baselines.}
  \label{tab:rq1-results}
  \begin{tabularx}{\textwidth}{@{}l c c c X@{}}
    \toprule
    \textbf{Condition} & \textbf{Precision (\%)} & \textbf{Recall (\%)} & \textbf{$F_1$-Score (\%)} & \textbf{Issues Found (out of 48)} \\
    \midrule
    axe-core Baseline   & 93.3 $\pm$ 0.0          & 29.2 $\pm$ 0.0       & 44.4 $\pm$ 0.0            & 14 \\
    Lighthouse Baseline & 88.9 $\pm$ 0.0          & 33.3 $\pm$ 0.0       & 48.5 $\pm$ 0.0            & 16 \\
    \versimux{} (Ours)  & 79.6 $\pm$ 3.4          & 81.3 $\pm$ 2.9       & 80.4 $\pm$ 3.1            & 39 $\pm$ 1.4 \\
    \bottomrule
  \end{tabularx}
\end{table}

The baseline static checkers (axe-core and Lighthouse) achieved high precision but low recall, identifying only static code violations (e.g., missing \code{alt} attributes or form fields lacking programmatic labels). They failed to detect interaction-level accessibility issues, such as the product modal focus trap or dynamic form alerts that were not announced to screen readers. 

In contrast, \versimux{} achieved a recall of 81.3\%, identifying 39 of the 48 expert-documented issues. The slightly lower precision (79.6\%) was primarily due to the persona agents flagging usability friction points (e.g., long lists lacking jump links) that, while not strict WCAG violations, were noted as valid design improvements by the expert panel. This outcome supports hypothesis $H1$.

% ---- 7.5 Persona Fidelity and Grounding (RQ2) -------------------------------
\section{Persona Fidelity and Grounding Effectiveness}
\label{sec:rq2-results}

To evaluate the impact of the grounding guards and trace verification mechanisms, we performed an ablation study comparing the full system configuration against two variants:
\begin{enumerate}
    \item \textbf{No-Grounding:} The persona agents generated actions (coordinates and CSS selectors) without validation.
    \item \textbf{No-Verification:} The agent trace verification node was bypassed, permitting raw, un-validated interaction traces to feed the downstream clustering engine.
\end{enumerate}

\Cref{tab:ablation-results} details the results of this ablation study.

\begin{table}[htbp]
  \centering
  \caption{Ablation Study: Grounding and Verification Impact.}
  \label{tab:ablation-results}
  \begin{tabularx}{\textwidth}{@{}l c c c X@{}}
    \toprule
    \textbf{Variant} & \textbf{Action Hallucinations (\%)} & \textbf{Trace Failures (\%)} & \textbf{F1-Score (\%)} & \textbf{Simulation Loop Rate (\%)} \\
    \midrule
    No-Grounding     & 36.4 $\pm$ 4.2                      & 42.0 $\pm$ 5.1               & 52.3 $\pm$ 4.6         & 24.8 $\pm$ 3.6 \\
    No-Verification  & 0.0 $\pm$ 0.0                       & 12.0 $\pm$ 2.4               & 71.4 $\pm$ 3.8         & 8.4 $\pm$ 1.2 \\
    \versimux{} (Full)& 0.0 $\pm$ 0.0                       & 2.0 $\pm$ 0.8                & 80.4 $\pm$ 3.1         & 0.0 $\pm$ 0.0 \\
    \bottomrule
  \end{tabularx}
\end{table}

Without grounding guards, the persona agents hallucinated CSS selectors in 36.4\% of actions, resulting in a 42\% trace failure rate due to browser script execution errors. In the full system configuration, the grounding guards caught and resolved these invalid actions prior to browser execution, resulting in a 0\% hallucination execution rate. 

Additionally, the trace verification node successfully detected and terminated infinite interaction loops (e.g., repeated scrolling). These findings support hypothesis $H2$.

% ---- 7.6 Patch Quality (RQ3) -----------------------------------------------
\section{Patch Quality}
\label{sec:rq3-results}

The patching pipeline's performance was evaluated based on patch applicability, verification success rate, and syntax compliance. The recommenders generated 42 patches across the five repetitions, categorized into HTML attributes (e.g., \code{aria-label}, \code{role}), CSS adjustments (e.g., color contrast, visible focus rings), and JavaScript handlers (e.g., keydown focus traps).

\Cref{tab:patch-results} summarizes the performance across these patch categories.

  \label{tab:patch-results}

    \midrule

    JavaScript Hooks    & 8                  & 62.5 $\pm$ 6.2        & 50.0 $\pm$ 7.1         & 100.0 $\pm$ 0.0 \\

HTML patches achieved the highest success rate, resolving 77.3\% of targeted issues. JavaScript patches (such as implementing focus trapping in modal panels) were more complex to apply, achieving a 50\% verification rate due to differences in page-specific DOM structures. 

All generated patches passed linter validation, ensuring syntax compliance. Overlapping patches were merged or resolved by the conflict resolver node, resulting in no compilation crashes. This outcome supports hypothesis $H3$.

\label{sec:rq4-results}

To evaluate the utility of \versimux{}'s output, we conducted a blind review study with 12 professional frontend software developers. Participants were provided with two reports for the same set of test pages: one generated by axe-core, and one generated by \versimux{}.

% ---- 7.8 Performance and Scalability ----------------------------------------
\section{Performance and Scalability}
\label{sec:performance}

To evaluate system performance, we measured pipeline latency, API token usage, and concurrency scaling.

\subsection{Pipeline Latency}
\label{sec:latency}

The end-to-end processing time for a single HTML page averaged 2.4 minutes under standard configuration. The persona simulation phase was the longest component, accounting for 62\% of the total latency. This is due to the sequential execution of actions within the browser context and the processing delay of the API endpoints.

\subsection{Token Consumption}
\label{sec:token-consumption}

An average evaluation run consumed 28,400 input tokens and generated 4,100 output tokens. The persona simulation phase consumed 72\% of the input tokens due to the injection of the pruned DOM tree and the history logs at each interaction step.

\subsection{Concurrency Scaling}
\label{sec:concurrency-scaling}

We evaluated system scaling by increasing the number of pages processed concurrently. Because the backend uses a \code{ThreadPoolExecutor} for browser contexts, processing throughput scaled linearly up to 4 concurrent worker threads without a significant increase in page-level latency. 

Beyond 4 threads, CPU scheduling delays and rate limits on the API key pool resulted in increased execution times, highlighting the importance of the rate-limiting queue subsystem.
