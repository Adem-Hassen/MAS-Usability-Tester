\chapter{Infrastructure and Observability}
\label{ch:infrastructure}

This chapter describes the runtime infrastructure that makes \versimux{} operational: the event-driven communication architecture, session management, real-time streaming, and frontend integration.


% ---- 6.1 Event Bus and Logging Bridge ---------------------------------------
\section{Event Bus and Logging Bridge}
\label{sec:event-bus}

At the core of \versimux{}'s runtime telemetry is the \code{EventBus} class, implemented as a thread-safe singleton. Because the multi-agent execution pipeline runs concurrently across multiple system threads managed by a \code{ThreadPoolExecutor}, the telemetry subsystem must support concurrent event registration and emission without causing database lockups or racing states. 

The \code{EventBus} handles this via a thread lock guard (\code{_lock = threading.Lock()}) that wraps instance acquisition. Telemetry consumers subscribe to the event stream by registering callbacks with the bus:

\begin{lstlisting}[language=Python, caption={Thread-Safe EventBus Implementation (core/event\_bus.py).}]
import threading
from typing import Callable, Optional

class EventBus:
    _instance: Optional["EventBus"] = None
    _lock = threading.Lock()
    
    def __init__(self):
        self._handlers: list[Callable] = []
    
    @classmethod
    def get(cls) -> "EventBus":
        with cls._lock:
            if cls._instance is None:
                cls._instance = cls()
            return cls._instance
    
    def subscribe(self, handler: Callable[[str, dict], None]):
        self._handlers.append(handler)
    
    def emit(self, event_type: str, **payload):
        for handler in self._handlers:
            try:
                handler(event_type, payload)
            except Exception:
                pass
\end{lstlisting}

To bridge standard logging output with the streaming telemetry interface, the backend configures a logging bridge processor. When structured log events are emitted by the agents (using the \code{structlog} library), they are intercepted by a custom processor. This processor checks the log messages for specific pipeline keys (e.g., \code{graph.supervisor_node.start}, \code{page_pipeline.start}, \code{persona.action}) and maps them directly to frontend step IDs and progress percentages. This architecture enables developers to view log lines and structured progress updates simultaneously.

\section{Server-Sent Events Streaming and State Recovery}
\label{sec:sse-streaming}

To feed the dashboard's real-time visual progress tracker, the backend exposes a \acr{SSE} stream via the `/api/v1/evaluate/{job_id}/stream` endpoint. Unlike traditional short-polling or WebSocket connections, SSE offers a unidirectional, low-overhead HTTP streaming channel that fits the event-driven progress logging model of the evaluation pipeline.

\subsection{Last-Event-ID Reconnection Protocol}
\label{sec:reconnection-protocol}

A primary challenge in streaming telemetry over long-running jobs is connection instability. If the client experiences a transient network drop, it will reconnect, but any events emitted during the downtime are lost, resulting in an desynchronised frontend state. 

\versimux{} resolves this by implementing the standard `Last-Event-ID` reconnection protocol. Each event emitted by the server is prefixed with an auto-incrementing integer ID. When a disconnected client re-establishes its connection, it sends its last successfully processed ID via the `Last-Event-ID` HTTP header. 

Upon receiving this header, the server queries the SQLite database to fetch and replay all buffered events whose IDs are greater than the client's last received ID, before seamlessly subscribing the client back to the active queue.

\subsection{Keepalive Heartbeat Mechanism}
\label{sec:keepalive}

To prevent intermediary proxies, firewalls, and reverse proxies (e.g., Nginx) from dropping idle connections during quiet periods of agent reasoning, the server implements a keep-alive heartbeat. If no telemetry events are pushed to the client's active queue for 15 seconds, the generator yields an empty comment string (\code{": keepalive\textbackslash{}n\textbackslash{}n"}), maintaining the connection without impacting the client's event handlers.


% ---- 6.3 Session Store and Persistence --------------------------------------
\section{Session Store and Persistence}
\label{sec:session-store}

The \code{SessionStore} class manages local persistence and coordination. It relies on a local SQLite database configured in Write-Ahead Logging (WAL) mode to permit concurrent read and write transactions.

\subsection{Database Schema}
\label{sec:db-schema}

The persistence model consists of two primary tables:
\begin{itemize}
    \item \textbf{\code{sessions}:} Tracks metadata for each execution session, including the primary ID, status (ready, running, done, failed), overall progress percentage, file counts, output paths, and the final results payload.
    \item \textbf{\code{events}:} Stores the complete sequential history of telemetry events. Each record tracks the global event ID, session ID, event kind, ISO timestamp, and a JSON payload containing details (e.g., action coordinates, screenshot references, or issues).
\end{itemize}

\subsection{Thread-Safe Event Dispatching}
\label{sec:thread-safe-dispatching}

Because database writes are triggered by background threads running the LangGraph agent pipeline, direct calls to asynchronous event queues would cause thread conflicts. To bridge this boundary, the \code{SessionStore} captures the main asyncio event loop reference during initialization (\code{self._loop = asyncio.get_running_loop()}). 

When an agent thread calls \code{store.emit()}, the store writes the event to the SQLite database and dispatches it to the active async queues using the thread-safe API:

\begin{code}
loop.call_soon_threadsafe(q.put_nowait, event)
\end{code}

This ensures that the web server remains responsive while agents execute tasks concurrently.

\subsection{Active Task Registry and Cancellation}
\label{sec:task-registry}

The store maintains an active task registry (\code{self._active_tasks}) mapping active session IDs to their running \code{asyncio.Task} handlers. If a user requests job cancellation (via the `/api/v1/evaluate/{job_id}/cancel` endpoint), the server looks up the task, invokes \code{task.cancel()}, and executes cleanup routines, including shutting down any remaining Playwright Chromium instances.


% ---- 6.4 Pipeline Runner ----------------------------------------------------
\section{Pipeline Runner}
\label{sec:pipeline-runner}

The \code{pipeline_runner.py} script coordinates the transition from FastAPI endpoint requests to active LangGraph execution. It wraps the execution within \code{asyncio.to_thread} to keep the main event loop unblocked.

\subsection{Telemetry Mapping and Progress Tracking}
\label{sec:telemetry-mapping}

The runner contains two configuration structures, \code{_STEP_MAP} and \code{_FUNC_STEP_MAP}, that map internal system events and python module executions to high-level UI progress indicators (e.g., mapping `clustering_node` to a 30\% progress indicator with the text "Clustering issues..."). 

A background heartbeat thread runs in parallel to monitor the pipeline's progress. If no log events are generated for 20 seconds, the thread emits a heartbeat progress indicator to prevent the frontend from timing out or entering an idle state.

\subsection{Screenshot Optimisation and Management}
\label{sec:screenshot-optimisation}

During simulation, the Playwright browser context captures screenshots of each step in base64 format. Passing these raw base64 strings directly through the SSE stream would cause significant token bloat and consume network bandwidth. 

To prevent this, the pipeline runner intercepts these images, decodes them, writes them to disk as compressed JPEGs under the session's output directory, and replaces the base64 payload with a relative asset URL (e.g., \code{/api/v1/evaluate/{job_id}/screenshots/{filename}}). These images are served by the backend's static file routing.


% ---- 6.5 Frontend Integration -----------------------------------------------
\section{Frontend Integration}
\label{sec:frontend}

The frontend is implemented in Next.js as a dashboard that coordinates state using React context (\code{PipelineContext}). The dashboard relies on a pure state reducer (\code{applyEvent}) to handle incoming SSE telemetries and update the UI:

\begin{code}
function applyEvent(prev: PipelineState, ev: StreamEvent): PipelineState {
  const next = { ...prev };
  switch (ev.kind) {
    case 'pipeline_start':
      next.status = 'running';
      next.progress = 2;
      break;
    case 'persona_action':
      next.livePreviews[ev.persona_id] = { ... };
      break;
    ...
  }
  return next;
}
\end{code}

This architecture ensures that the frontend remains decoupled from the backend's internal structures. The client consumes a single stream of events, applying changes incrementally. If the connection fails, the client uses an exponential backoff retry strategy to reconnect, sending its last received event ID to resume the stream without losing progress data.


% ---- 6.5.1 Nexus Dashboard UI Layout -----------------------------------------
\subsection{Nexus Dashboard UI Layout}
\label{sec:dashboard-layout}

To manage the high density of concurrent event telemetry, the \versimux{} user interface is designed as a three-panel responsive layout, referred to as the Nexus Dashboard. This layout is structured to maintain distinct operational boundaries between process tracking, live simulation feedback, and remediation output:

\begin{description}
  \item[Pipeline Rail (Left Panel).] A fixed-width sidebar (256\,px) that acts as the primary process supervisor indicator. It contains a vertical stepper tracking the active LangGraph execution node (e.g., supervisor, persona simulation, clustering, recommender, conflict resolver, verification, and reporting). The rail integrates a live progress bar and animated counters that update the current count of detected issues and proposed patches in real time using a custom \code{useAnimatedNumber} React hook.
  
  \item[Live Preview Panel (Center Panel).] A flexible viewport displaying live execution screenshots streamed from Playwright browser contexts. It supports a multi-column responsive grid (1 to 3 columns) for parallel swarm observation, and a full-width focus carousel with manual navigation. Using the bounding box coordinates (\code{\{x, y, width, height\}}) received in telemetry payloads, the panel overlays semi-transparent SVG rectangles directly on top of the screenshots. These rectangles represent user actions (such as clicks, mouse movements, and text entries) and include animated ripples to visually highlight the target element. Individual persona preview cards are decorated with dynamic border colors determined by hashing the persona's identifier (\code{persona_id}) to prevent visual confusion between active agents.
  
  \item[Output and Remediation Console (Right Panel).] A dedicated tabbed workspace (500\,px) that enables developers to toggle between inspecting raw issue cards, browsing proposed patch catalogs, reviewing code-level diffs, and downloading final reports.
\end{description}

% ---- 6.5.2 Interactive Code-Diff and Remediation Viewer ---------------------
\subsection{Interactive Code-Diff and Remediation Viewer}
\label{sec:code-diff-viewer}

The primary mechanism for patch validation is an interactive diff viewer built directly into the remediation console. Rather than presenting static unified patches, the viewer provides a rich side-by-side comparison interface:

\begin{itemize}
  \item \textbf{Myers Diff Engine:} The client implements a TypeScript-based version of the Myers Longest Common Subsequence (\acr{LCS}) algorithm to compute line-level insertions and deletions dynamically. A secondary LCS pass is run on modified lines to compute word-level changes, highlighting character insertions and deletions inside a line to accelerate developer review.
  
  \item \textbf{Syntax Tokenization:} To render the compared code blocks without loading heavy external highlighting libraries, the client uses a lightweight regular-expression-based HTML tokenizer. This custom tokenizer parses HTML elements, separating tags (rendered in rose), attributes (rendered in amber), and string values (rendered in emerald).
  
  \item \textbf{Context Folding:} Code files often contain large sections of unchanged markup. The viewer automatically collapses contiguous lines of identical code that are further than 3 lines away from a diff region, substituting them with collapsible \code{···} separators. This focuses developer attention entirely on the proposed edits.
\end{itemize}

% ---- 6.5.3 End-to-End Developer Workflow ------------------------------------
\subsection{End-to-End Developer Workflow}
\label{sec:developer-workflow}

The operational workflow for a developer utilizing \versimux{} to audit and fix an interface consists of four sequential stages:

\begin{enumerate}
  \item \textbf{Upload and Config.} The developer uploads 1--5 HTML files via the dashboard's landing page. They can optionally specify natural-language task guidelines, desired target goals (e.g., "submit the registration form successfully"), and configuration parameters such as the maximum persona budget or correction loop iterations.
  
  \item \textbf{Live Telemetry Observation.} Upon initiating the execution, the backend starts the LangGraph runner. The developer watches the Live Preview Panel, observing how the persona agents navigate the pages, where they experience accessibility friction, and what issues they flag. The console below the preview streams raw server logs to maintain transparency.
  
  \item \textbf{Patch Audit.} Once the simulation and recommender swarm conclude execution, the proposed patches are displayed. The developer reviews the edits inside the side-by-side Diff Viewer, assessing the exact HTML, CSS, or JavaScript modifications.
  
  \item \textbf{Export.} After completing the verification loops, the developer downloads a unified ZIP package containing the fully patched HTML, CSS, and JS files, alongside a comprehensive PDF diagnostic report outlining WCAG compliance details, persona feedback, and resolution success rates.
\end{enumerate}
